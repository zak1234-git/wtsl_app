# 协议与数据流

## 1. 协议代码位置总览
- HTTP：`src/wtsl_core/protocol/http/`
- HTTPS：`src/wtsl_core/protocol/https/wtsl_core_httpsmain.c`（空）
- MQTT：`src/wtsl_core/protocol/mqtt/wtsl_core_mqttmain.c`（仅返回 `-1`）
- Modbus：`src/wtsl_core/protocol/modbus/wtsl_core_modbusmain.c`（空）
- CAN：`src/wtsl_core/protocol/can/wtsl_core_canmain.c`（空）
- RS485：`src/wtsl_core/protocol/rs485/wtsl_core__rs485main.c`（空）

## 2. HTTP 处理流程（已确认主路径）
### 2.1 GET 路径
1. `request_handler` 收到 GET 请求并进入 `http_get_cmd`。  
2. `http_get_cmd` 调 `wtcoreapi_parse_http_get_cmd(url, upload_data, size)`。  
3. `wtcoreapi_parse_http_get_cmd` 解析 URL，命中 `nodes` 后调用 `wtsl_core_get_nodes_functions`。  
4. `wtsl_core_get_nodes_functions` 通过 `CALL_INTERFACE` 调节点回调（如 `get_node_basicinfo/get_node_conninfo/...`）。  
5. 业务返回 `APIRET` 后由 `data_response_to_json` 封装，再 `MHD_queue_response` 回包。

证据函数：
- `src/wtsl_core/protocol/http/wtsl_core_httpmain.c::request_handler/http_get_cmd`
- `src/wtsl_core/protocol/http/wtsl_core_http_restful_api.c::wtcoreapi_parse_http_get_cmd/wtsl_core_get_nodes_functions`
- `src/wtsl_core/dataparser/wtsl_core_dataparser.c::data_response_to_json`

### 2.2 POST 路径
1. `request_handler` 分段累积 `upload_data` 到 `request_data.data`。  
2. 数据接收完成后调 `wtsl_core_http_deal_post_data`。  
3. 进入 `wtcoreapi_parse_http_post_cmd`。  
4. 命中 `nodes` 资源后进 `wtsl_core_post_nodes_functions`。  
5. 按 URL 子路径执行 `CALL_INTERFACE`（如 `set_node_basicinfo/do_connect/do_upload/do_upgrade`）。  
6. 使用 `data_response_to_json` 统一响应。

## 3. 节点同步协议（UDP 心跳）
### 3.1 数据结构
- `NodeDataHeader { TAG, type, time, reserv }`
- 消息类型：`MSG_HEARTBEAT_REQ/RSP/COMMAND/RESPONSE`

证据：`src/wtsl_core/slal/node/wtsl_core_node_service.c`。

### 3.2 流程
1. `heartbeat_thread` 周期构造目标地址并发送心跳请求。  
2. `recv_thread` 读取 UDP 报文，`parse_data` 按 `header.type` 分发。  
3. `parse_data_heartbeat` 将 `SPLINK_INFO` 同步进节点链表（新增或更新）。  
4. 对 `HEARTBEAT_REQ` 回复 `send_heartbeat_rsp`。

## 4. 组播同步（G 节点）
1. `wtsl_core_splink_gnode_service` 初始化组播收发 socket。  
2. 周期发送 `HEARTBEAT:<ts>`。  
3. `recv_heartbeats` 收包并 `update_node_list`。  
4. `clean_timeout_mc_nodes` 清理超时节点。

证据：`src/wtsl_core/slal/wtsl_core_splink_mc.c`。

## 5. 三条“外部请求→内部处理→响应/结果”典型调用链
### 链路 A：查询单节点基础信息（HTTP GET）
- 外部：`GET /api/v1/nodes/<id>/basicinfo`
- 内部：
  1) `request_handler`
  2) `http_get_cmd`
  3) `wtcoreapi_parse_http_get_cmd`
  4) `wtsl_core_get_nodes_functions`
  5) `CALL_INTERFACE(..., get_node_basicinfo, ...)`
  6) `wtsl_core_get_node_basicinfo`
  7) `data_response_to_json`
- 结果：返回包含节点基础参数的 JSON。

### 链路 B：修改节点基础参数（HTTP POST）
- 外部：`POST /api/v1/nodes/<id>/basicinfo` + JSON body
- 内部：
  1) `request_handler` 累积请求体
  2) `wtsl_core_http_deal_post_data`
  3) `wtcoreapi_parse_http_post_cmd`
  4) `wtsl_core_post_nodes_functions`
  5) `CALL_INTERFACE(..., set_node_basicinfo, ...)`
  6) `wtsl_core_set_node_basicinfo`
  7) `wtsl_core_parse_json_setnode`
  8) `set_node`（下发参数）+ `config_set*`（持久化）
  9) `data_response_to_json`
- 结果：返回 `setnode_success`（被包装在成功 JSON 中）。

### 链路 C：固件升级（HTTP POST）
- 外部：
  - 上传：`POST /api/v1/nodes/<id>/firmware/upload`
  - 升级：`POST /api/v1/nodes/<id>/firmware/upgrade`
- 内部：
  1) 路由到 `wtsl_core_post_nodes_functions`
  2) `CALL_INTERFACE(..., do_upload, ...)` -> `wtsl_core_do_upload` -> `wtsl_core_parse_json_extract_file_header`
  3) `CALL_INTERFACE(..., do_upgrade, ...)` -> `wtsl_core_do_upgrade` -> `wtsl_core_firmware_upgrade`
  4) `parse_firmware_header` + `extract_and_verify_firmware` + `handle_firmware_upgrade`
- 结果：返回升级受理/失败 JSON。

## 6. 已确认 / 推断 / 待确认
### 已确认
- HTTP 是当前已落地的外部管理协议主链路。
- UDP/组播用于节点同步，不直接暴露为 REST。

### 推断
- MQTT/HTTPS/Modbus/CAN/RS485 目录可能是未来扩展协议入口，当前分支尚未启用。

### 待确认
- 各空协议目录是否在其他分支已有实现，或在产品配置中按宏裁剪。

## 7. `wtsl_core_http_restful_api.c` 函数级阅读手册

代码位置：`src/wtsl_core/protocol/http/wtsl_core_http_restful_api.c`

本节目标：按“每个函数做什么”来读该文件，并标注“重点函数”和“辅助函数”。

### 7.1 文件中的核心类型与全局变量
1. `UrlData`：URL 分段结构体，仅保存一个子串 `str`。
2. `WTApiFuncPtr`：统一函数指针签名，`(url, args, size) -> void*`。
3. `WTAPI_FUNC_MAP`：路由映射表项，包含 `apiname + pfunc`。
4. `g_url_list`：保存本次 URL 分段后的链表，供分发函数按索引读取。

### 7.2 函数总览（按重要度）

#### A. 重点函数（主路径）
1. `wtcoreapi_parse_http_get_cmd`：GET 总入口，拆 URL、命中资源映射、调用资源处理函数。
2. `wtcoreapi_parse_http_post_cmd`：POST 总入口，流程同上但传递请求体数据。
3. `wtsl_core_get_nodes_functions`：`/api/v1/nodes/...` GET 分发函数。
4. `wtsl_core_post_nodes_functions`：`/api/v1/nodes/...` POST 分发函数。

#### B. 资源映射相关（重要）
1. `wtapi_get_func_mappings`：GET 资源映射数组。
2. `wtapi_post_func_mappings`：POST 资源映射数组。

#### C. 辅助函数（当前主路径影响较小）
1. `param_arr`：URL 切分调试函数，当前主流程未调用。
2. `node_id_compare`：节点 ID 比较函数，当前文件主流程未直接调用。

#### D. 旧接口兼容函数（`#ifdef OLD_API`，默认关闭）
1. `wtsl_core_self_test`
2. `wtsl_core_view_users`
3. `wtsl_core_tnode_join_net`
4. `wtsl_core_tnode_show_bss`
5. `wtsl_core_scan`
6. `wtsl_core_setnode`
7. `wtsl_core_get_hw_resources_usage`
8. `wtsl_core_uploadfirmware`
9. `wtsl_core_upgrade_firmware`

说明：默认构建路径中 `OLD_API` 未开启，这些函数通常不参与线上主链路。

### 7.3 重点函数详解

#### 7.3.1 `wtcoreapi_parse_http_get_cmd`
作用：处理 GET 请求的第一站路由。

实现步骤：
1. 初始化返回值 `APIRET ret = {status=-1, data=NULL}`，先假定失败。
2. 创建 `url_list`，并用 `strtok(url, "/")` 把 URL 拆段写入链表。
3. 做合法性检查：分段数量不足（小于 3）直接返回不支持。
4. 遍历 `wtapi_get_func_mappings`，构造形如 `/api/v1/<apiname>` 的前缀进行匹配。
5. 命中后调用映射函数 `pfunc(url, upload_data, upload_data_size)`。
6. 若返回非空，置 `ret.status=0`，返回上层。

为什么这样做：
1. 把“资源选择”统一放在一处，便于扩展新资源。
2. 将具体业务下沉到资源函数，避免入口变成超大 if-else。

#### 7.3.2 `wtcoreapi_parse_http_post_cmd`
作用：POST 请求第一站路由，整体结构与 GET 对称。

实现步骤：
1. 同样初始化 `APIRET` 失败值。
2. 拆 URL 到 `g_url_list`。
3. 遍历 `wtapi_post_func_mappings` 做资源级匹配。
4. 命中后调用 `pfunc(url, data, size)`。
5. 返回 `status/data` 给 HTTP 接入层。

与 GET 的主要差异：
1. 参数是请求体 `data`，不是 query 上传数据。
2. 主要用于配置修改、动作触发等写操作。

#### 7.3.3 `wtsl_core_get_nodes_functions`
作用：处理 `GET /api/v1/nodes/...` 的“资源内二级分发”。

实现思路：
1. 先确认资源段是 `nodes`。
2. 如果 URL 长度为 3（例如 `/api/v1/nodes`），走“获取所有节点”。
3. 如果 URL 长度为 5（例如 `/api/v1/nodes/<id>/basicinfo`），取出：
  - `id_str = g_url_list[3]`
  - `func_str = g_url_list[4]`
4. 通过 `find_wtsl_node_by_id` 找到目标节点。
5. 依据 `func_str` 进入不同分支（`basicinfo/conninfo/advinfo/scan/...`）。
6. 每个分支通过 `CALL_INTERFACE` 调用节点接口层。

为什么这样做：
1. 路由层只负责“选函数”，不写具体业务。
2. `CALL_INTERFACE` 统一走权限检查和调用包装，安全边界更清晰。

#### 7.3.4 `wtsl_core_post_nodes_functions`
作用：处理 `POST /api/v1/nodes/...` 的二级分发。

实现思路：
1. 要求 URL 至少 5 段（否则判定不支持）。
2. 提取 `api_str/id_str/func_str`，定位目标节点。
3. 按 `func_str` 分支执行 `set/connect/reboot/restore/...` 等动作。
4. 固件升级路径 `firmware/upload|upgrade` 继续读取第 6 段进行三级分发。
5. 调用同样通过 `CALL_INTERFACE` 进入节点接口层。

为什么这样做：
1. 将“同一资源下的不同动作”集中管理，便于统一审计。
2. 支持多级路径（如 firmware/upload），扩展性好于平铺 URL。

### 7.8 核心函数实现细节（变量级）

#### 7.8.1 `wtcoreapi_parse_http_get_cmd` 的完整执行过程
输入：`url`、`upload_data`、`upload_data_size`。

执行顺序：
1. 先构造失败默认值：`ret.status=-1`，`ret.data=NULL`。
2. 创建 URL 分段链表：`List *url_list = list_create(sizeof(UrlData))`。
3. 把全局 `g_url_list` 指向这个 `url_list`，供后续分发函数按索引读取。
4. `tmp_url = strdup(url)` 复制一份可写 URL，因为 `strtok` 会改写字符串。
5. 循环 `strtok(tmp_url, "/")`，每个片段再 `strdup(token)` 后插入链表。
6. 调试输出各索引内容，确认 URL 分段正确。
7. 长度保护：当 `url_list->size < 3` 时直接返回不支持。
8. 遍历 GET 资源映射数组 `wtapi_get_func_mappings`。
9. 每轮构造资源前缀 `geturl = /api/v1/<apiname>`。
10. 取第 3 段 `api_name_in_url = g_url_list[2]` 做资源名匹配。
11. 命中后调用映射函数 `pfunc(url, upload_data, upload_data_size)`。
12. 若 `pfunc` 返回非空，置 `ret.status=0`，并返回给上层。

关键点：
1. 这是“资源级路由”，不负责 `basicinfo/conninfo` 这种动作级路由。
2. 动作级路由在 `wtsl_core_get_nodes_functions` 内继续处理。

#### 7.8.2 `wtcoreapi_parse_http_post_cmd` 的完整执行过程
输入：`url`、`data`、`size`。

执行顺序：
1. 同 GET，先构造默认失败 `APIRET`。
2. 同样创建并填充 `g_url_list`。
3. 遍历 `wtapi_post_func_mappings` 做资源匹配。
4. 命中后调用 `pfunc(url, data, size)`。
5. `pfunc` 非空则置成功，返回上层。

和 GET 的差异：
1. 这里传的是 POST 请求体，通常用于设置和动作触发。
2. 匹配条件使用前缀匹配，允许资源后面挂更多路径段。

#### 7.8.3 `wtsl_core_get_nodes_functions` 的完整执行过程
输入：`url`、`args`、`size`。

执行顺序：
1. 定义静态输出缓冲区 `static char recv_bss[1024]`。
2. 读取 `api_str = g_url_list[2]`，必须是 `nodes`。
3. 若 URL 长度 `<=3`，按“查询全部节点”处理：
  - 取节点列表
  - 用 `id=0` 找本地节点
  - 调 `CALL_INTERFACE(..., get_all_nodes_info, ...)`
4. 若 URL 长度 `==5`，按“单节点动作查询”处理：
  - `id_str = g_url_list[3]`
  - `func_str = g_url_list[4]`
  - `find_wtsl_node_by_id(atoi(id_str))`
5. 根据 `func_str` 分支：
  - `basicinfo` -> `get_node_basicinfo`
  - `conninfo` -> `get_node_conninfo`
  - `advinfo` -> `get_node_advinfo`
  - `scan` -> `start_scan`
  - 以及 `resourcesinfo/show_bss_info/sle_*` 等
6. 每个分支都通过 `CALL_INTERFACE` 进入节点接口层。
7. 任一分支失败返回 `NULL`，成功返回结果指针。

关键点：
1. `CALL_INTERFACE` 不是直接调用业务，它会先做权限检查。
2. URL 的“资源、节点 ID、动作名”三要素都在这里解析。

#### 7.8.4 `wtsl_core_post_nodes_functions` 的完整执行过程
输入：`url`、`args`、`size`（`args` 一般是 JSON 文本）。

执行顺序：
1. 校验 URL 分段长度，至少需要 `api/v1/nodes/<id>/<func>`。
2. 读取：
  - `api_str = g_url_list[2]`
  - `id_str = g_url_list[3]`
  - `func_str = g_url_list[4]`
3. 定位目标节点 `find_wtsl_node_by_id(atoi(id_str))`。
4. 进入动作分支并调用对应接口：
  - `basicinfo` -> `set_node_basicinfo`
  - `advinfo` -> `set_node_advinfo`
  - `connect/reboot/restore/...`
  - `sle_*` 系列动作
5. `firmware` 为三级路径：继续读取 `g_url_list[5]` 判断 `upload/upgrade`。
6. `size==6` 的额外分支处理 `traffic` 路径。
7. 任一分支失败返回 `NULL`，成功返回业务结果指针。

关键点：
1. 这是动作执行最密集的分发函数，后续加 POST API 通常在这里改。
2. 所有动作依然统一走 `CALL_INTERFACE`，保持权限与调用行为一致。

### 7.9 阅读该文件时优先关注的实现风险
1. `g_url_list` 是全局变量，多请求并发时要关注线程安全与覆盖风险。
2. URL 分段时多次 `strdup`，需要关注释放路径是否完整。
3. 资源匹配和动作匹配大量使用前缀比较，新增 API 时应避免歧义命名。
4. `static` 返回缓冲区在并发请求下可能被后续请求覆盖。

## 8. 下一层：`node_manager`（CALL_INTERFACE 实现层）

代码位置：
1. `src/wtsl_core/slal/node/wtsl_core_node_manager.h`
2. `src/wtsl_core/slal/node/wtsl_core_node_manager.c`
3. `src/wtsl_core/slal/node/wtsl_core_node_callback.h`

这一层的职责：
1. 定义“节点接口表”和权限映射。
2. 提供统一调用入口 `interface_wrapper`。
3. 在真正调用业务函数前完成参数校验、权限校验、本地/远端分流。

### 8.1 为什么路由层只写一行 `CALL_INTERFACE`

路由层调用示例：
1. `CALL_INTERFACE(p_node, get_node_basicinfo, recv_bss, sizeof(recv_bss), &g_user_ctx)`

展开后会进入：
1. `node->interface->get_node_basicinfo`（拿到真实函数指针）
2. `FUNC_get_node_basicinfo`（拿到该函数对应权限索引）
3. `interface_wrapper(...)`（统一做权限和分流）

所以路由层不直接调回调函数，是为了把“安全控制”和“调用规范”统一到一处。

### 8.2 头文件里最关键的三块定义

#### 8.2.1 `NODE_FUNC_LIST`（X 宏清单）
作用：维护所有可被路由层调用的节点动作。

每一项包含 4 个元素：
1. 接口名（如 `get_node_basicinfo`）
2. 所需权限（如 `PERMISSION_READ`）
3. 实际函数实现名（如 `wtsl_core_get_node_basicinfo`）
4. 索引号（如 `1`）

意义：
1. 通过一份清单同时生成“接口表、枚举索引、权限表”。
2. 避免多处手写导致不一致。

#### 8.2.2 `WTSLNodeInterface`
作用：节点接口函数指针集合。

它不是业务实现本身，而是函数入口表。每个节点对象持有一个 `interface` 指针，路由层可通过它调用动作。

#### 8.2.3 `CALL_INTERFACE` 宏
作用：把函数名转换成：
1. 对应函数指针
2. 对应功能索引
3. 统一交给 `interface_wrapper`

这样所有调用都强制经过同一个“门禁”。

### 8.3 `node_manager.c` 的函数级说明

#### 8.3.1 `default_interface`（全局接口表）
作用：把 `NODE_FUNC_LIST` 中声明的函数实现，批量挂到接口表上。

实现特点：
1. 使用 X 宏展开自动生成字段赋值。
2. 每个字段最终是 `WTSLNodeCallBack*`。

结果：节点对象默认就有一整套可调用动作。

#### 8.3.2 `global_perm_map`（全局权限映射）
作用：把函数名与所需权限、索引绑定起来。

实现特点：
1. 同样通过 X 宏从 `NODE_FUNC_LIST` 自动生成。
2. 保证“接口函数”和“权限定义”天然一致。

#### 8.3.3 `check_is_remote_node`
作用：判断目标节点是本地节点还是远端节点。

实现逻辑：
1. 比较目标节点 MAC 与本机 `global_node_info` 的 MAC。
2. 不一致则判定远端（返回 0）。
3. 一致则判定本地（返回 -1）。

#### 8.3.4 `call_remote_node_cmd`
作用：远端节点调用预留入口。

现状：
1. 当前仅打日志并返回 `NULL`。
2. 说明远端调用链在该分支还未完整落地。

#### 8.3.5 `interface_wrapper`（本层最核心函数）
作用：统一包装所有节点接口调用。

输入：
1. `node`：目标节点
2. `real_func`：真实业务函数指针
3. `func_index`：功能索引（用于查权限）
4. `data/size`：业务参数
5. `ctx`：用户上下文（权限主体）

执行顺序：
1. 参数合法性校验：任一关键参数为空直接失败。
2. 从 `node->perm_map[func_index]` 取该动作所需权限。
3. 调 `check_permission_user(ctx, required)` 做权限校验。
4. 若权限不足，记录日志并返回 `NULL`。
5. 调 `check_is_remote_node` 判断本地/远端。
6. 远端则走 `call_remote_node_cmd`（当前返回 NULL）。
7. 本地则直接执行 `real_func(node, data, size, ctx)`。

为什么这样实现：
1. 权限校验与业务实现解耦，避免每个 callback 重复写权限判断。
2. 本地与远端调用可在同一入口分流，便于未来扩展远程执行链路。

### 8.4 `node_callback.h` 在这层中的作用

该文件本质是“接口实现声明清单”，用于告知：
1. `NODE_FUNC_LIST` 中的函数实现在哪里。
2. 每个动作的函数签名统一为 `void* func(void* pNode, void* data, unsigned int size, UserContext* ctx)`。

对阅读者意义：
1. 路由层看到 `CALL_INTERFACE(..., do_connect, ...)` 后，可直接在该头文件找到对应实现函数名，再跳转到 callback.c 阅读细节。

### 8.5 从 REST 路由到 callback 的完整穿透链

以 `POST /api/v1/nodes/0/connect` 为例：
1. `wtcoreapi_parse_http_post_cmd` 命中 `nodes`。
2. 进入 `wtsl_core_post_nodes_functions`，命中 `func_str=connect`。
3. 调 `CALL_INTERFACE(p_node, do_connect, args, size, &g_user_ctx)`。
4. 宏展开进入 `interface_wrapper(...)`。
5. `interface_wrapper` 做权限校验并判定本地/远端。
6. 本地路径调用 `wtsl_core_do_connect(...)`（callback 实现）。
7. 结果返回上层封装为 HTTP JSON。

### 8.6 新增动作时这一层需要改什么

当新增一个动作（示例：`do_xyz`）：
1. 在 `wtsl_core_node_callback.h` 增加函数声明。
2. 在 `wtsl_core_node_callback.c` 实现函数。
3. 在 `NODE_FUNC_LIST` 增加 `X(do_xyz, 权限, 实现函数, 新索引)`。
4. 路由层分支里用 `CALL_INTERFACE(p_node, do_xyz, ...)` 调用。

这样可以自动获得：
1. 接口表挂载
2. 权限映射
3. 统一 wrapper 流程

### 8.7 本层阅读时优先关注的风险点
1. `call_remote_node_cmd` 尚未实现，远端调用路径当前可能不可用。
2. 权限索引依赖 `NODE_FUNC_LIST` 顺序，手工改索引需谨慎。
3. `interface_wrapper` 的空指针检查是最后防线，新增节点初始化流程要确保 `interface/perm_map` 已正确赋值。

### 9.2 重点函数实现细节（变量级）

#### 9.2.1 `wtsl_core_set_node_basicinfo`
作用：设置节点基础参数并同步全局状态/日志文件名。

执行步骤：
1. 校验 `phandle` 和 `data` 非空。
2. 调 `wtsl_core_parse_json_setnode(data)` 解析 JSON 并下发参数。
3. 解析成功后按新配置重命名日志文件（`generate_file_name + rename`）。
4. 调 `update_wtsl_node_basicinfo(...)` 将本地节点状态同步到节点链表。
5. 返回固定成功字符串 `setnode_success`。

实现要点：
1. 该函数同时影响配置、链表状态、日志文件路径。
2. 返回值是短字符串，不是完整 JSON；真正 JSON 包装在上层完成。

#### 9.2.2 `wtsl_core_set_node_advinfo`
作用：设置节点高级参数（MIB/能力等）并同步链表。

执行步骤：
1. 校验节点句柄和数据。
2. 调 `wtsl_core_parse_json_set_node_advinfo(data)`。
3. 成功后 `update_wtsl_node_basicinfo(...)` 同步链表。
4. 返回 `set_advinfo_success`。

#### 9.2.3 `wtsl_core_get_node_basicinfo`
作用：读取单节点基础信息并返回 JSON 字符串。

执行步骤：
1. 用 `create_node_info_json(pNode)` 组装 cJSON 对象。
2. `cJSON_Print` 得到字符串。
3. 检查调用方缓冲区大小是否足够。
4. `strncpy` 复制到 `data`，返回 `data` 指针。

#### 9.2.4 `wtsl_core_get_node_advinfo`
作用：读取高级信息并返回 JSON。

执行步骤：
1. 调 `create_node_advinfo_json`。
2. 函数内部会 `sync_advinfo()` 后 `sleep(1)` 再组装字段。
3. 输出到 `data` 缓冲区返回。

注意：
1. 该路径包含同步和等待，响应时延会高于普通查询。

#### 9.2.5 `wtsl_core_do_time_sync`
作用：校验并设置系统时间。

执行步骤：
1. 解析 JSON，读取 `time` 字段。
2. 用 `is_valid_time` 做格式和合法性校验。
3. 合法时执行 `system("date -s ...")`。
4. 成功后重命名日志文件路径。
5. 返回 `set time success`。

#### 9.2.6 `wtsl_core_do_connect`
作用：执行入网连接动作。

执行步骤：
1. 打印目标节点信息。
2. 调 `wtsl_core_parse_json_tnode_join_net(data)`。
3. 成功返回 `{"status":"connected"}` 字符串。

#### 9.2.7 `wtsl_core_do_disconnect`
作用：按 MAC 踢掉用户。

执行步骤：
1. 从 JSON 取 `mac`。
2. 组装 `iwpriv ... kick_user <mac>` 命令并 `popen` 执行。
3. 解析命令输出中的状态（如 `SUCC`）。
4. 组装返回 JSON：`success` 或 `failed`。

#### 9.2.8 `wtsl_core_do_reboot`
作用：校验时间/token 权限后重启设备。

执行步骤：
1. 解析 `username/time/token`。
2. 先校验时间合法性。
3. 再调 `check_user_has_permission(...)`。
4. 通过后执行 `system("reboot")`。

#### 9.2.9 `wtsl_core_do_upload` 与 `wtsl_core_do_upgrade`
作用：OTA 上传与升级触发。

`do_upload`：
1. 把请求体二进制写入 `/tmp/upgrade.bin`。
2. 调 `wtsl_core_parse_json_extract_file_header` 解析包头。

`do_upgrade`：
1. 直接对 `/tmp/upgrade.bin` 调 `wtsl_core_firmware_upgrade`。

#### 9.2.10 `wtsl_core_do_chswitch`
作用：修改信道并下发驱动命令。

执行步骤：
1. 解析 JSON 的 `channel`。
2. 做白名单校验（允许值集合）。
3. `config_set_int("CHANNEL", channel)` 持久化。
4. `iwpriv ... bss_chan_switch` 下发切信道。

#### 9.2.11 `wtsl_core_do_aifh`
作用：设置 ACS/AIFH 开关。

执行步骤：
1. 解析 `aifh_sw`。
2. 校验值必须是 0/1。
3. 持久化 `AIFH_ENABLE`。
4. 调 `iwpriv ... set_acs_enable`。

#### 9.2.12 `wtsl_core_get_traceinfo`
作用：按 `feature` 分发到具体 trace 子函数。

执行步骤：
1. 解析 JSON，读取 `user_mac` 和 `feature`。
2. 根据 `feature` 选择：
  - `vip`
  - `throughput`
  - `delay`
  - `drop`
  - `mcs`
  - `user_access`
3. 调对应 `wtsl_core_get_traceinfo_*` 子函数。

子函数共性：
1. 执行 `iwpriv ... trace ...`。
2. 解析多行文本统计值。
3. 组装结构化 JSON 返回。

#### 9.2.13 SLE 相关函数
1. `wtsl_core_sle_start_scan`：触发扫描，停止 seek，再复用 show_bss 返回结果。
2. `wtsl_core_sle_show_bss`：遍历扫描结果数组，输出 `rssi/mac` JSON 数组。
3. `wtsl_core_sle_connect`：解析 `mac` 后构造 `sle_addr_t` 并发起连接。
4. `wtsl_core_sle_conninfo`：遍历连接管理表，返回当前连接列表。
5. `wtsl_core_get_sle_basicinfo`：返回本机 SLE 基础字段。
6. `wtsl_core_set_sle_basicinfo`：设置 `sle_name/sle_type`，并持久化；类型变化时会重启。
7. `wtsl_core_set_announce_id`：解析 announce_id 数组写入全局数组。

#### 9.2.14 测试模式函数

### 10.2 重点函数实现细节（变量级）

#### 10.2.1 `wtsl_core_parse_json_setnode`
作用：解析基础配置 JSON，写入全局节点参数，下发并持久化。

执行步骤：
1. 解析 JSON 根对象，失败直接返回。
2. 在互斥锁保护下读取字段并写入：`type/channel/bw/tfc_bw/cell_id/name/ip/password/log_port/net_manage_ip`。
3. 解析复合字段：`set_mib_params`、`set_mcs_bound`。
4. 调 `set_node()` 下发参数到驱动/设备层。
5. 通过 `config_set_int/config_set` 持久化关键字段到配置文件。
6. 释放 JSON，解锁，返回成功。

关键点：
1. 这是“解析+下发+持久化”一体函数。
2. 由 callback 层调用后，会再做节点链表同步。

#### 10.2.2 `wtsl_core_parse_json_set_node_advinfo`
作用：解析高级参数并下发。

执行步骤：
1. 读取 `cell_id/cc_spos/cp_type/symbol_type/sysmsg_period/s_cfg_idx/sec_exch_cap/range_opt/acs_enable/tx_power`。
2. 写入 `global_node_info` 的高级结构体。
3. 调 `set_node()` 执行实际生效。
4. 将高级字段写入配置项持久化。

关键点：
1. 该函数会改动 MIB 与能力相关参数。
2. 对应 callback 是 `wtsl_core_set_node_advinfo`。

#### 10.2.3 `wtsl_core_parse_json_tnode_join_net`
作用：解析入网参数并执行入网流程。

说明：
1. 被 `wtsl_core_do_connect` 调用。
2. 返回值用于判断连接动作成功或失败。

#### 10.2.4 `wtsl_core_parse_json_tnode_show_bss`
作用：解析/整理 BSS 扫描结果到输出缓冲区。

说明：
1. 被扫描类接口复用。
2. 通常与 `wtsl_core_parse_json_scan` 配合使用。

#### 10.2.5 `wtsl_core_parse_json_scan`
作用：触发扫描动作。

说明：
1. 常见调用顺序是先 scan，再 show_bss。
2. callback 层会在扫描后 sleep 一段时间再取结果。

#### 10.2.6 `wtsl_core_parse_json_extract_file_header`
作用：处理上传包头，拆出升级元信息并回传。

执行步骤：
1. 检查上传文件是否存在、大小是否合法。
2. 读取头部数据并尝试解析固件 JSON 头。
3. 解析成功后输出头信息日志。
4. 从上传文件中提取 tar 区段写入临时升级文件。
5. 返回包含版本、类型、MD5、文件名的成功 JSON 字符串。

关键点：
1. 该函数返回 `strdup` 字符串，调用方需有释放策略。
2. 解析失败时会返回标准错误 JSON 字符串。

#### 10.2.7 `wtsl_core_firmware_upgrade`
作用：固件升级总流程。

执行步骤：
1. 读取升级包头并解析为 `firmware_header_t`。
2. 做平台架构校验与 MD5 校验。
3. 调 `handle_firmware_upgrade` 执行升级。
4. 根据组件存在情况（app/www/firmware/lib）拼接不同成功响应。
5. 返回最终 JSON 字符串。

关键点：
1. 该函数流程长，包含大量失败分支和错误信息返回。
2. 是 OTA 最核心函数，建议单独重点阅读。

#### 10.2.8 `data_response_to_json`
作用：把上层 `status + data` 封装成统一响应。

当前行为：
1. `status != 0` -> 输出 `{status: Failed}`。
2. `status == 0` -> 输出 `{status: success, data: <parsed-json>}`。

注意点：
1. 当 `data` 不是合法 JSON 时，`cJSON_Parse(data)` 可能失败。
2. 函数内部有动态分配，调用链需统一释放返回字符串。

### 10.3 本层与上下层的协作关系
1. 上层 callback 负责“动作入口与流程控制”。
2. dataparser 负责“JSON 字段落地与参数下发”。
3. 更底层 slb/sle 接口负责“具体驱动命令执行”。

### 10.4 本层阅读风险点
1. 解析失败路径很多，错误码和返回字符串需要统一审计。
2. 某些键名存在不一致风险（如配置项命名），改动需双向核对读写路径。
3. 升级路径大量字符串拼接与文件操作，需重点检查边界与内存管理。
4. 统一响应函数对非法 JSON 输入的兼容性需重点关注。

## 11. 下一层：`slb_interface`（设备执行与驱动下发层）

代码位置：
1. `src/wtsl_core/slal/slb/wtsl_core_slb_interface.h`
2. `src/wtsl_core/slal/slb/wtsl_core_slb_interface.c`

这一层的职责：
1. 把上层参数转换为真实设备命令（大量 `ifconfig/iwpriv/system/popen`）。
2. 提供“设置/读取”设备参数的统一函数（`slb_set_* / slb_get_*`）。
3. 负责节点初始化、默认参数加载后的首次下发。
4. 提供高级信息同步函数 `sync_advinfo`，把设备实时状态回填到内存结构。

### 11.1 启动路径中的关键入口

从核心启动函数 `wtsl_core_main` 的调用关系看：
1. `param_default()`：先给全局节点结构体填默认值。
2. `config_read()`：再用配置文件覆盖默认值。
3. `init_node()`：把当前内存参数真正下发到设备并起网口。

说明：这是“默认值 -> 配置覆盖 -> 设备生效”的三段式启动。

### 11.2 重点函数实现细节（变量级）

#### 11.2.1 `param_default`
作用：给 `global_node_info` 初始化一套可启动的默认参数。

执行内容：
1. 生成随机 MAC 并写入节点结构。
2. 设置默认 `bw/tfc_bw/channel/type/ip/name/log_port`。
3. 设置高级参数默认值：`cell_id/cc_offset/mib_params/sec_exch_cap/...`。
4. 设置网桥默认接口列表与数量。
5. 设置 SLE 默认类型与名称。

关键点：
1. 该函数只改内存，不直接下发设备。
2. 后续会被 `config_read` 覆盖，再由 `init_node/set_node` 生效。

#### 11.2.2 `init_node`
作用：系统启动时执行一次完整初始化和首次参数下发。

执行步骤：
1. 准备 firmware 目录并复制资源到目标目录。
2. 加载内核模块（支持两套模块路径策略）。
3. 创建 SLB 设备并设置 MAC。
4. 启动用户态服务（如 `uspd`）。
5. 关闭网口后依次下发关键参数：
  - `channel`
  - `bw`
  - `tfc_bw`
  - `mcs_bound`
  - `cc_offset`
  - `cell_id`
  - `essid`
  - `mib_params`
  - `tx_power`
  - `sec_auth_pwd`
6. G 节点场景附加下发 `acs_enable`。
7. 初始化网桥并最终拉起网口。

关键点：
1. 这是“冷启动全量下发”函数。
2. 依赖大量系统命令，失败处理多为日志告警。

#### 11.2.3 `set_node`
作用：运行期参数重下发入口（配置修改后常走这里）。

执行步骤：
1. `slb_ifconfig(...,0)` 先 down 网口。
2. 下发 `channel`。
3. 按当前与目标 `bw` 关系调整 `bw/tfc_bw` 下发顺序。
4. 下发 `mcs_bound/cc_offset/cell_id/essid/mib_params/password/sec_exch_cap/range_opt`。
5. G 节点下发 `acs_enable`。
6. 初始化网桥并拉起网口。
7. 同步 IP 到前端配置文件（`sync_ip_to_js_cfg`）。
8. 更新设备编号 `devid` 与发射功率。
9. 检查 `TYPE` 是否发生 G/T 切换，若切换则重启。

关键点：
1. callback/dataparser 改参数后，真正“设备生效”主要靠它。
2. 该函数有重启分支，联调时需谨慎。

#### 11.2.4 `sync_advinfo`
作用：从设备侧读取高级状态，回填到 `global_node_info`。

执行步骤：
1. 读取并回填 `cc_offset/cell_id/devid/chip_temperature/range_opt/acs_enable`。
2. 读取并回填 `mib_params`。
3. 读取并回填安全能力：`sec_exch_cap/sec_sec_cap/lce_mode/pps_enable`。
4. 读取并回填统计结构：`timestamp/wds/user_mcs/real_power/rssi_data`。
5. 读取并回填 `fem_check` 字符串状态。

关键点：
1. 高级信息查询接口（如 advinfo）会依赖该同步结果。
2. 失败路径多为日志告警，字段可能部分更新。

### 11.3 `slb_set_* / slb_get_*` 家族函数模式

#### 11.3.1 设置类函数共同模式（以 `slb_set_channel/slb_set_bw/slb_set_tfc_bw` 为例）
1. 校验设备名合法。
2. 校验参数取值范围（如信道白名单、带宽只能 20/40/80）。
3. 通常先 down 网口。
4. 组装 `iwpriv` 命令并 `system` 执行。
5. 返回 0/-1。

#### 11.3.2 读取类函数共同模式（以 `slb_get_channel/slb_get_bw` 为例）
1. 校验设备名。
2. `popen` 执行 `iwpriv ... get_*`。
3. 逐行读取输出并解析值（常见按 `=` 分隔）。
4. 返回整数结果。

#### 11.3.3 复杂设置函数（示例）
1. `slb_set_mib_params`：多参数组装命令，失败点较多。
2. `slb_set_msc_bound`：校验 MAC 与 MCS 范围后下发。
3. `slb_set_tx_power`：设置发射功率并校验返回状态。

### 11.4 本层与上层关系
1. dataparser 负责“把 JSON 字段变成结构体参数”。
2. slb_interface 负责“把结构体参数变成设备命令并执行”。
3. callback 层负责“业务流程编排”。

### 11.5 本层阅读风险点
1. 命令输出解析强依赖文本格式，驱动输出变化会破坏解析。
2. 设置函数大量 `system/popen`，错误码语义不完全统一。
3. 参数白名单（尤其信道）硬编码，升级频点时需同步修改。
4. `set_node` 含重启逻辑，类型切换会触发设备重启。
