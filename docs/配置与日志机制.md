# 配置与日志机制

## 1. 配置机制

## 1.1 配置入口与文件
- 运行时配置文件路径常量：`ENV_CONFIG = /home/wt/env_config`
- 配置读写 API：`config_get/config_get_int/config_set/config_set_int`
- 启动时读取：`wtsl_core_main -> config_read`

证据：
- `src/wtsl_core/slal/slb/wtsl_core_slb_interface.h::ENV_CONFIG`
- `src/wtsl_core/utils/wtsl_cfg_manager.c::config_* / config_read`
- `src/wtsl_core/wtsl_core.c::wtsl_core_main`

## 1.2 配置加载流程（步骤）
1. `param_default` 写入一套默认值到 `global_node_info`。  
2. `config_read` 用配置值覆盖默认值。  
3. `init_node`/`set_node` 将当前值下发到设备。  
4. 当接口修改参数时，通过 `config_set*` 回写配置文件。

## 1.3 线程安全
- 配置文件读写受 `file_mutex` 保护。  
证据：`src/wtsl_core/utils/wtsl_cfg_manager.c::file_mutex`。

## 2. 日志机制

## 2.1 初始化与输出路径
- 初始化：`wtsl_logger_init(APP_NAME, global_node_info)`
- 日志文件：默认 `/home/wt/log/<node_name>__YYYY-MM-DD.log`（由 `generate_file_name` 生成）
- 宏入口：`WTSL_LOG_DEBUG/INFO/WARNING/ERROR/FATAL`

证据：
- `src/wtsl_core/wtsl_core.c::wtsl_core_main`
- `src/wtsl_core/utils/wtsl_log_manager.c::wtsl_logger_init/generate_file_name/wtsl_logger_log`
- `src/wtsl_core/include/wtsl_log_manager.h`

## 2.2 日志级别与模式
- 级别：DEBUG/INFO/WARN/ERROR/FATAL
- 模式：
  - `CONFIG_MODE_ALL`：全部记录
  - `CONFIG_MODE_PRODUCT`：只记录 `WARNING/ERROR/FATAL`
- 模式配置来源：`/home/system.conf` 中 `[chip log_system]` 节

证据：`src/wtsl_core/utils/wtsl_log_manager.c::parse_config_file/wtsl_logger_should_log`。

## 2.3 动态重载
- `need_config_reload` 定时检查配置文件 mtime。
- 变化时触发 `wtsl_logger_reload_config`。

## 2.4 输出介质
1) 控制台（带颜色）  
2) 日志文件  
3) 达到大小阈值后由上传线程触发 TCP 上传

证据：
- 控制台+文件：`wtsl_logger_log`
- 上传触发：`wtsl_client_upload.c::need_upload_and_clear`、上传线程相关逻辑（由日志系统初始化创建线程）

## 3. 日志与配置的交互边界
- 配置模块只做 KV 存储，不做日志格式策略。
- 日志模块读取独立配置（`/home/system.conf`），不依赖 `ENV_CONFIG`。
- 业务模块通过日志宏输出，不直接操作日志文件句柄。

## 4. 关键日志点（建议重点关注）
- 启动阶段：`wtsl_core_main`（版本、参数、服务启动）
- HTTP 入口：`request_handler`（方法、URL、数据长度）
- 路由分发：`wtcoreapi_parse_http_*`
- 节点参数变更：`wtsl_core_set_node_basicinfo`、`wtsl_core_parse_json_setnode`
- 升级流程：`wtsl_core_parse_json_extract_file_header`、`wtsl_core_firmware_upgrade`

## 5. 已确认 / 推断 / 待确认
### 已确认
- 日志系统具备“按模式过滤 + 配置热更新 + 上传线程”三项机制。
- 配置写入采用“临时文件 + rename”策略，降低写坏风险。

### 推断
- 在生产模式下，INFO 日志大量被过滤，排障应优先切到 ALL 模式。

### 待确认
- 日志上传线程完整实现位于 `wtsl_client_upload.c` 后半段，建议结合运行日志确认触发条件与服务器协议对接是否一致。
