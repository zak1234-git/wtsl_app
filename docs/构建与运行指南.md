# 构建与运行指南

## 1. 先理解构建产物
项目将 `src/wtsl_core` 编译为库，再链接生成主程序：
- 静态库：`lib/<arch>/libwtsl_core.a`
- 动态库：`lib/<arch>/libwtsl_core.so`
- 可执行文件：`bin/<arch>/wtsl_app`

证据：`Makefile` 目标 `$(WTCORE_STATIC_LIB)`、`$(WTCORE_SHARED_LIB)`、`$(TARGET)`。

## 2. 依赖与环境
### 2.1 编译工具链
- 默认架构 `ARCH=arm`
- 交叉编译前缀：`arm-mix510-linux-`（arm）/ `aarch64-none-linux-gnu-`（arm64）

证据：`Makefile` 中 `ARCH` 和 `CROSS_COMPILE` 分支。

### 2.2 链接依赖
- `pthread`
- `cjson`
- `microhttpd`
- `wtsl_core`
- `sle_host_*`（按架构）

证据：`Makefile` 中 `LDLIBS`。

### 2.3 运行期系统依赖（目标机）
代码中大量调用：`iwpriv`、`ifconfig`、`udhcpc`、`udhcpd`、`killall`、`httpd`、`nanddump` 等。  
证据：
- `src/wtsl_core/slal/slb/wtsl_core_slb_interface.c`
- `src/wtsl_core/slal/node/wtsl_core_node_service.c`
- `src/wtsl_core/utils/wtsl_cfg_manager.c`
- `src/wtsl_core/dataparser/wtsl_core_dataparser_ota.c`

## 3. 标准构建命令
- 默认构建：`make`
- 指定架构：`make ARCH=arm64` / `make ARCH=x64`
- 仅构建 core 库：`make wtcore-libs ARCH=arm`
- 清理：`make clean` 或 `make clean-arm`

证据：`Makefile` 的 `all/wtcore-libs/clean/help`。

## 4. 运行入口与启动行为
### 4.1 运行入口
- 可执行入口：`src/wtsl_app/main.c::main`
- 主流程：`main -> wteapp_main -> wtsl_core_main`

### 4.2 启动时关键行为（顺序）
1. 初始化默认参数：`param_default`
2. 读取配置文件：`config_read`
3. 初始化硬件/网络节点：`init_node`
4. 初始化日志系统：`wtsl_logger_init`
5. 启动 HTTP 服务：`wtsl_http_main`（宏开启时）
6. 创建状态与服务线程：`create_splink_state_watcher`

证据：`src/wtsl_core/wtsl_core.c::wtsl_core_main`。

## 5. 常见构建问题（附定位建议）
### 问题 A：预编译脚本报错
- 现象：`make` 在 `prebuild` 阶段失败。
- 已确认：`Makefile` 调用了 `cfg/PreBuild.sh`，但工作区中未找到该文件；存在的是 `src/wtsl_core/cfg/PreBuild.sh`。
- 建议：
  1) 补齐 `cfg/PreBuild.sh`；或
  2) 修改 `Makefile` 预编译路径。

### 问题 B：库缺失
- 现象：`check-libs` 阶段报 `未找到libcjson/libmicrohttpd`。
- 证据：`Makefile::check-libs`。
- 建议：将库放入 `lib/` 或 `lib/<arch>/`。

### 问题 C：运行后接口无响应
- 排查顺序：
  1) 看 `HTTP_PORT` 配置是否存在（`config_get("HTTP_PORT",...)`）
  2) 看 `MHD_start_daemon` 返回值
  3) 看日志模式是否过滤（`/home/system.conf` 中 `[chip log_system]`）

证据：`src/wtsl_core/protocol/http/wtsl_core_httpmain.c::wtsl_http_main`、`src/wtsl_core/utils/wtsl_log_manager.c::parse_config_file/wtsl_logger_should_log`。

## 6. 已确认 vs 推断
### 已确认
- 构建系统会将 `libwtsl_core.so` 和部分第三方 so 复制到 `bin/<arch>/`。
- 程序退出时会停 HTTP 并释放核心资源。

### 推断
- 项目更偏向在目标板运行（非 PC 原生运行），x64 主要用于开发期编译验证。

- `src/wtsl_core/dataparser/wtsl_core_dataparser.c` 当前内容是否已通过本分支编译。
