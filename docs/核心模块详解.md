# 核心模块详解

> 阅读方式：每个模块都按“职责 → 关键函数 → 上下游 → 何时被调用 → 输入输出”说明。

## 1) core（启动编排层）
- 代码位置：`src/wtsl_core/wtsl_core.c`
- 职责：统一初始化各子系统，管理进程运行/退出。
- 关键函数：
  - `wtsl_core_main`：启动编排。
  - `wtsl_core_release`：停止标志置位 + 杀掉相关进程。
- 上游：`src/wtsl_app/main.c::wteapp_main`
- 下游：`param_default/config_read/init_node/wtsl_logger_init/wtsl_http_main/create_splink_state_watcher`
- 输入输出：
  - 输入：全局参数、配置文件内容。
  - 输出：完成服务启动，返回 `ret`。

## 2) api（核心状态采集层）
- 代码位置：`src/wtsl_core/api/wtsl_core_api.c`
- 职责：提供版本信息与链路状态采集。
- 关键函数：
  - `wtsl_core_get_version`：填充版本号与 commit 短 ID。
  - `wtsl_core_get_splink_info`：汇总 `view_users/channel/version/rssi` 并更新 `SPLINK_INFO`。
- 上游：`wtsl_core_main`、`wtsl_core_splink_state_watcher`
- 下游：`slb_get_channel`、`get_view_users_info`、`get_rssi_info`、`config_set_int`
- 输入输出：输入 `SPLINK_INFO*`，输出链路状态 `link_state`。

## 3) dataparser（参数解析与响应组装）
- 代码位置：
  - `src/wtsl_core/dataparser/wtsl_core_dataparser.c`
  - `src/wtsl_core/dataparser/wtsl_core_dataparser_ota.c`
- 职责：
  - 解析 HTTP JSON 请求到内部参数
  - 将业务结果包装成统一 JSON 响应
  - 处理 OTA 上传、校验、升级执行
- 关键函数：
  - `wtsl_core_parse_json_setnode`
  - `wtsl_core_parse_json_set_node_advinfo`
  - `data_response_to_json`
  - `wtsl_core_parse_json_extract_file_header`
  - `wtsl_core_firmware_upgrade`
- 上游：节点回调、HTTP 主流程
- 下游：`set_node`、`config_set*`、`handle_firmware_upgrade`
- 输入输出：
  - 输入：URL 对应 JSON 体 / 二进制升级包
  - 输出：状态码 + JSON 字符串

## 4) protocol（协议接入层）
### 4.1 HTTP
- 代码位置：`src/wtsl_core/protocol/http/`
- 职责：接收 REST 请求，路由到节点接口，返回 JSON。
- 关键函数：`wtsl_http_main`、`request_handler`、`wtcoreapi_parse_http_get_cmd/post_cmd`。
- 输入输出：输入 HTTP 请求；输出 HTTP JSON 响应。

### 4.2 MQTT
- 代码位置：`src/wtsl_core/protocol/mqtt/wtsl_core_mqttmain.c`
- 已确认：`wtsl_mqtt_main()` 仅 `return -1`。

### 4.3 HTTPS / Modbus / CAN / RS485
- 代码位置：`src/wtsl_core/protocol/https|modbus|can|rs485/`
- 已确认：多个 `*main.c` 为空文件（当前分支未实现）。

## 5) slal（设备抽象与节点同步层）
### 5.1 slb 接口
- 代码位置：`src/wtsl_core/slal/slb/wtsl_core_slb_interface.c`
- 职责：封装设备参数下发、驱动模块处理、网络桥接初始化。
- 关键函数：`param_default`、`init_node`、`set_node`。

### 5.2 node 管理
- 代码位置：`src/wtsl_core/slal/node/`
- 职责：
  - 节点对象与链表管理（`wtsl_core_node_list.c`）
  - 权限控制 + 接口包装（`wtsl_core_node_manager.c`）
  - 业务回调实现（`wtsl_core_node_callback.c`）
  - UDP 心跳服务（`wtsl_core_node_service.c`）
- 关键函数：`interface_wrapper`、`find_wtsl_node_by_id`、`parse_data_heartbeat`、`recv_thread`。

### 5.3 splink 状态与组播
- 代码位置：`src/wtsl_core/slal/wtsl_core_splink_state.c`、`wtsl_core_splink_mc.c`
- 职责：状态刷新、多线程服务编排、组播心跳同步。
- 关键函数：`create_splink_state_watcher`、`wtsl_core_splink_state_watcher`、`wtsl_core_splink_gnode_service`。

## 6) utils（基础能力层）
- `wtsl_cfg_manager.c`：配置读写、配置文件并发保护。
- `wtsl_log_manager.c`：日志文件输出、模式切换、上传线程。
- `wtsl_client_upload.c`：日志文件打包成帧并通过 TCP 上传。
- `wtsl_core_net_utils.c`：网卡 IP 获取与默认网关推导。

## 7) user（权限模型）
- 代码位置：`src/wtsl_core/user/user_manager.c`
- 职责：提供 `UserContext` 和位掩码权限检查。
- 关键函数：`check_permission_user`
- 当前默认用户：`g_user_ctx = {1001, PERMISSION_ADMIN}`。

## 8) 模块边界（日志/配置/网络）
- 配置模块边界：只负责 KV 读写 + 配置注入，不直接处理 HTTP。
- 日志模块边界：对外提供统一 `WTSL_LOG_*` 宏，不关心具体业务路由。
- 网络/协议模块边界：HTTP 负责接入，节点回调负责业务，SLB 负责设备参数执行。

## 9) 已确认 / 推断 / 待确认
### 已确认
- 模块调用边界总体清晰：`protocol -> node interface -> dataparser/slb`。
- 权限检查发生在 `interface_wrapper`，不是在 HTTP 层。

### 推断
- 项目正在从“单协议控制”向“多协议接入”演进，但目前生产路径仍以 HTTP 为主。

### 待确认
- 远程节点调用 `call_remote_node_cmd` 当前返回 `NULL`，尚未见完整远程执行链路。

## 10) 风险点（新手必须关注）
1. 高耦合全局状态：`global_node_info` 被多个线程和模块读写，虽然有互斥，但跨模块依赖强。  
  证据：`inc/wtsl_core_api.h::global_node_info`、`src/wtsl_core/slal/wtsl_core_splink_state.c`、`src/wtsl_core/api/wtsl_core_api.c`。
2. 外部命令依赖重：大量 `system/popen` 调系统命令，平台迁移成本高。  
  证据：`src/wtsl_core/slal/slb/wtsl_core_slb_interface.c`、`src/wtsl_core/dataparser/wtsl_core_dataparser_ota.c`。
3. 协议实现不均衡：HTTP 完整，其它协议目录存在空文件或占位函数。  
  证据：`src/wtsl_core/protocol/mqtt/wtsl_core_mqttmain.c`、`https/modbus/can/rs485` 目录文件。
4. 错误处理路径不统一：部分函数只返回 `NULL/-1`，上下文错误码语义不一致。  
  证据：`wtsl_core_http_restful_api.c`、`wtsl_core_node_callback.c` 多处错误分支。
5. 当前分支存在可维护性风险：个别源码出现疑似异常文本，影响可编译性与阅读准确性。  
  证据：`src/wtsl_core/protocol/http/wtsl_core_http_restful_api.c`、`src/wtsl_core/dataparser/wtsl_core_dataparser.c`。
